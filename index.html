CropVision AI

Crop Disease Detection Platform

GitHub ↗ Demo: To be updated

A crop-disease workflow that connects YOLOv8 inference to authenticated scans, validated image uploads, stored detections, and scan history.

Image upload→FastAPI + YOLOv8→Express API→PostgreSQL history

Read full case study +

Problem

A model prediction alone does not give an agricultural user a usable way to submit images, receive structured detections, or revisit earlier scans.

Motivation

Explore how computer vision can support crop monitoring by turning detections into a structured, reviewable software workflow.

My role

Built or contributed to the FastAPI YOLOv8 service, image validation, bounding-box outputs, authenticated Express workflow, PostgreSQL result persistence, Docker setup, and prediction/upload tests.

Solution

The FastAPI service validates images and returns detections. The Express layer protects scan routes with JWT, persists uploaded files, and stores samples and detection boxes in a PostgreSQL transaction.

Current result

The repository supports local multi-service operation, inference integration, protected scan submission, physical image persistence, and per-user scan history. Current status: in progress.

Limitations

Needs broader real-world crop data, clearer model evaluation, user testing, and a public deployment. The supplied repository does not establish field impact or production use.

Next steps

Strengthen dataset and evaluation reporting, improve the scan workflow, monitor inference failures, and prepare screenshots and a short demo.

Problem

A driver-monitoring prototype should keep its local workflow available during intermittent connectivity instead of treating the network as a hard dependency.

Motivation

Explore how camera input, lightweight on-device inference, local event handling, and delayed synchronization fit within ESP32-CAM constraints.

My role

The CV records work on ESP-DL face detection, dynamic 96×96 eye-region preprocessing, INT8 FOMO inference with TensorFlow Lite Micro and PSRAM, sustained-closure filtering, UART warnings, Telegram alerts, and a FreeRTOS upload task.

Solution

The CV describes an offline eye-state pipeline with local warning logic. The repository directly shows OV2640 capture plus SD-card JSONL logging, transaction-like file states, streaming upload, rollback, and network backoff.

Current result

Academic prototype. The repository demonstrates local camera operation and a resilient store-and-forward logging path; the full drowsiness pipeline is documented in the CV and design reports.

Limitations

Not a certified safety product. The supplied source does not provide controlled accuracy, latency, memory, lighting, or face-angle evaluation, and does not fully expose every CV-listed inference component.

Next steps

Consolidate the inference firmware into the public repository, measure latency and memory use, build a controlled eye-state evaluation set, and test low-light and off-angle behavior.

WSL-DR

Weakly Supervised Diabetic Retinopathy Grading

Problem

Detailed lesion annotations are expensive. Weak supervision investigates whether image-level labels can support grading while still producing lesion-aware visual explanations.

Motivation

Study medical-image model design and interpretation in a setting where annotation detail is limited.

My role

Implemented the five-class PyTorch grading pipeline, EfficientNet-B0 attention module, Swin Transformer backbone, PRCF feature fusion, Grad-CAM outputs, and QWK evaluation.

Solution

An EfficientNet attention module produces a lesion prompt, PRCF fuses it with Swin patch features, and the remaining Swin stages produce a grade. Grad-CAM supports inspection of model attention.

Current result

The repository records a validation QWK of 0.6942 and includes a CLI demo for grade prediction and four-panel attention visualization.

Limitations

Research implementation only. It needs stronger reproducibility and external validation; attention maps are supportive visualizations, not clinical evidence. Not for diagnosis.

Next steps

Document dataset preparation, add experiment tracking, compare backbones, publish a clearer evaluation report, and validate on an external dataset.
